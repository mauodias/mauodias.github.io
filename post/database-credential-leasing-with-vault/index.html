Database Credential Leasing With Vault

12 Mar 2019 on devops automation security database vault credential leasing

One of the biggest concerns of reasonably large projects and applications is the management of secrets. We have them laying everywhere, and there’s not much to do about it. We may need access tokens, passwords, credentials and much more. How should we handle those secrets? What is the best way to move such data around, to store them and to use, without chances of leaking by accident (or even intentionally) and, if such an incident happens, what are good ways to minimize the damage? This article will show one approach to mitigate some of these issues.

Please note that this approach is way far from being a silver bullet. There’s no perfect security scheme, so being extremely careful is never enough, but always needed.

The use case

In the client I’m currently working on, an interesting approach for secret management was developed: we have a repository packed with tools, scripts and encrypted secrets. Those secrets can only be edited through one of these tools, which relies on Amazon KMS, so the only users that will be allowed to encrypt and decrypt such secrets are the ones with enough access to a specific AWS account. Such access is provided through a IdaaS provider (Identity As A Service, definitions from Auth0, Okta, Ping Identity and BMC), so it is very hard to obtain unauthorized access to this account.
